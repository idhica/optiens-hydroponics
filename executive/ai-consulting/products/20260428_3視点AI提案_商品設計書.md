# 3視点AI提案 — 商品設計書

## 概要

stock-sun.comの「社内コンペ制度」（複数のWebコンサルタントが同案件にコンペ提案する仕組み）に着想を得て、Optiensの一人法人体制でも「複数の異なる視点からのAI活用提案を1度のヒアリングで提示する」を商品化する。Optiensは複数のClaude Codeエージェントを使い分けることで、人的に複数提案者を抱える大手と同等の「視点の多様性」を、一人法人の機動力で提供できる。これは現時点でOptiensにしかできない差別化軸である。

---

## 第1部：商品概要

### 商品名（決定）
**「3視点AI提案」**（正式名）

選定理由:
- 「マルチパース・AIプロポーザル」: カタカナ過多で意味が伝わらない
- 「AIトリプル診断」: 「診断」は無料診断と用語衝突するため不可（用語マスタ準拠）
- 「3視点AI提案」: 数字+漢字で直感的に内容が伝わる。「提案」は用語マスタ「提案書」と整合

### ポジショニング
- 既存4階層ファネルの「詳細レポート（¥5,000）」と「導入支援（個別見積）」の中間に位置する**追加オプション**
- 「詳細レポートだけでは決めきれないが、導入支援契約までは踏み込めない」層への中間商品
- Pattern Aの詳細レポート（¥5,000）の**上位アップグレード**として位置づける

### 想定顧客
- 中小企業の経営者・経営企画担当（従業員10-100名規模）
- 自社のAI活用について複数の選択肢を比較検討したい層
- 詳細レポート（¥5,000）受領後、導入支援に進む前にもう一段階ほしいと感じる層
- 商工会経由のPattern C-soft顧客で、温度感が中程度のケース

### 価格・期間
- **価格**: ¥30,000（税別）
- **納期**: 5〜7営業日
- **納品形態**: Google Slides（URL共有・9〜12枚）+ 60分オンラインMTG
- **MTG**: デフォルトで含む（オプトアウト不可。3視点を口頭で説明する性質上、MTG必須）

### 既存4階層への組み込み位置

```
無料レポート（¥0）
    ↓
詳細レポート + MTG（¥5,000）
    ↓ ← ここに「3視点AI提案（¥30,000）」を中間オプションとして配置
導入支援（個別見積）
    ↓
保守（月額）
```

Pattern Bの「¥30,000有償要件定義」とは別商品として運用する。
- ¥30,000有償要件定義: 直接導入支援を希望する顧客向け。1視点・実装前提
- 3視点AI提案: 比較検討段階の顧客向け。3視点・意思決定支援

価格が同額（¥30,000）の理由は、いずれも「Optiensの本気の工数を投下する」境界線がこの金額に置かれているため。差は「実装前提か比較検討か」の用途差である。

## 第2部：3視点の定義

### 採用する3視点

| 視点 | 担当エージェント仮想ロール | 中心問い | 主要KPI |
|---|---|---|---|
| **業務効率化視点** | COO/CFO目線 | 「どの業務を自動化すれば最も時間とコストが減るか」 | 削減時間・人件費削減額・ROI回収期間 |
| **顧客体験視点** | CMO目線 | 「どの接点をAI化すれば顧客満足と売上が伸びるか」 | 応答時間短縮・成約率・LTV |
| **データ活用視点** | CTO/CSO目線 | 「散在するデータを統合して何が見えるか・何ができるか」 | データ可視化・将来予測・意思決定速度 |

### 視点間の重複と差別化

**意図的な重複**:
- 同じ業務（例: 顧客問合せ対応）が複数視点で取り上げられる場合がある
- これは「同じ事象を別の角度で見ると優先度が変わる」ことを示すための設計
- 顧客は3視点を比較することで、自社の経営優先順位（コスト削減か売上拡大か基盤整備か）を再認識できる

**差別化のポイント**:
- 各視点は**独立した提案書**として読めること（他視点を読まなくても完結）
- 推奨アクションは視点ごとに**異なる順序**で提示する（同じ施策でも優先度が違う）
- 最終ページで3視点を一覧化し、顧客が選択判断できるサマリ表を提示

### 業種適用性

**汎用適用可能**: 業務効率化視点・顧客体験視点・データ活用視点の3軸は、ほぼ全業種で成立する。

**業種別カスタマイズが必要なポイント**:
- 業務効率化視点: 業種固有の繁忙業務が異なる（士業=書類作成、EC=受注処理、飲食=シフト管理など）
- 顧客体験視点: 顧客接点チャネルが異なる（B2B=メール/商談、B2C=LINE/SNS、自治体=窓口/電話）
- データ活用視点: 保有データの種類が異なる（製造=センサーデータ、EC=購買履歴、士業=案件履歴）

→ 業種別の標準テンプレート（士業/EC/飲食/製造/自治体の5業種）を整備し、ヒアリング内容で差し込む設計とする。

### 各エージェントへのプロンプト設計（骨子）

各エージェントには共通の入力（顧客フォーム回答+添付ファイル）を渡し、視点ごとに異なる役割プロンプトで提案を生成する。

**業務効率化視点エージェント**:
- 役割: COO/CFO視点で業務時間とコストの削減を最優先する提案者
- 出力フォーマット: 業務TOP3 / 各業務の月間削減時間・削減金額 / 自動化アーキテクチャ / 投資回収期間
- 推奨5-7件のうち、必ず削減時間が大きい順で並べる

**顧客体験視点エージェント**:
- 役割: CMO視点で顧客接点の質向上と売上拡大を最優先する提案者
- 出力フォーマット: 顧客接点TOP3 / 各接点のAI化案 / 応答時間・成約率・LTVへの定性影響 / 競合差別化
- 推奨5-7件のうち、必ず売上インパクトが大きい順で並べる

**データ活用視点エージェント**:
- 役割: CTO/CSO視点でデータ統合と意思決定基盤の構築を最優先する提案者
- 出力フォーマット: 散在データの一覧 / 統合構成図 / ダッシュボード設計案 / 将来の拡張余地
- 推奨5-7件のうち、必ず基盤投資→運用の順で並べる

**監修プロンプト（CEO実行）**:
- 3視点で重複している施策を抽出
- 視点間で矛盾している前提（例: 同じツールを別の用途で推奨している等）を統合
- 最終ページのサマリ表（3視点×推奨施策×優先度）を生成

## 第6部：差別化ポイント

### なぜOptiensにしかできないか

1. **AIエージェント運用基盤を持つ**: Claude Code + MCP + Supabaseの基盤を自社で運用済み（novel-review・社内CRMで実証）。視点別エージェントを切り替えて並行実行する仕組みは一般的なコンサル会社にはない
2. **一人法人の機動力**: 大手コンサルが複数人材で行う作業を、AIエージェントの切り替えで代替できる。意思決定者が1人なのでスケジュール調整・社内合意形成のオーバーヘッドがない
3. **カスタムUI構築力**: 顧客が継続利用する場合、提案内容を顧客専用ダッシュボードに実装する展開も可能（Supabase + Astroパターン）

### 競合との比較

| 項目 | 大手コンサル | 中小AI支援事業者 | Optiens 3視点AI提案 |
|---|---|---|---|
| 複数視点の提案 | 可能（複数人材必要） | 困難（単一視点） | 可能（エージェント切替） |
| 価格帯 | 数百万円〜 | 数十万円〜 | ¥30,000 |
| 納期 | 1-3ヶ月 | 2-4週間 | 5-7営業日 |
| 担当者の一貫性 | 複数担当（引継ぎコスト） | 1人 | 1人（CEOが全視点を統合） |
| カスタムUI実装への接続 | 別案件・別会社 | 困難 | 同一会社で連続提供可 |

### 顧客視点でのメリット

- **意思決定の質向上**: 1視点だけの提案より、複数視点の比較で経営判断の納得感が高まる
- **コストパフォーマンス**: 大手コンサルの複数視点提案（数百万円）と同等の構造を¥30,000で得られる
- **時間短縮**: 複数のコンサル会社に並行依頼する手間が不要
- **充当制度**: 導入支援に進む場合は¥30,000を初期費用に充当でき、無駄が出ない

### 採用しない訴求

業務フロー定義・用語マスタ・CLAUDE.md準拠で以下は使用しない:
- 「複数のAIが議論する」「AI同士のディベート」などの誇張表現
- 「業界初」「日本初」などの最上級表現
- 「先着」「今月限定」などのFOMO訴求
- 具体的な技術固有名詞（Claude Code・Supabase等）の社外発信での露出
- 大手コンサルへの直接的な対立フレーム（「大手より安い」など）

- 添付資料: 業務フロー定義v1.2

**3視点での仮想出力**:
- 業務効率化視点: 詳細レポート自動生成 / freee入金確認自動化 / カレンダー連動MTG準備
- 顧客体験視点: 無料レポート後の3日後フォローメール改善 / MTG前自動リマインド / 失注3ヶ月後再アプローチの個別化
- データ活用視点: Supabase CRMでの案件パイプライン可視化 / 配信ログ分析 / Phase別KPIダッシュボード

これらは既に業務フロー定義v1.2で部分的に設計されているため、Phase 1検証では「設計済みの内容を3視点で再構成できるか」のテストとして機能する。
