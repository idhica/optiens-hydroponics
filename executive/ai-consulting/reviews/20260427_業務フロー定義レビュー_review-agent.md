# 業務フロー定義レビュー（内部整合性・抜け漏れ・品質）

## 冒頭サマリ
v1.1への改訂でPattern Bのスクリーニングルールはメモリ定義と概ね整合した。ただし「Notion CRM予定」「frontmatter status: v1.0」など改訂取りこぼしが残る。重大なのはPattern AとPattern Bでの「導入支援への進入」前提の齟齬（Pattern Aは詳細レポートMTG内クロージング、Pattern Bは別途提案MTG）と、無償サポート3ヶ月の言葉の揺れ（implementation.astro「不具合修正」と業務フロー定義「不具合修正」は一致するが、定義書では「機能追加・改善は対象外」と明記される一方サイトでは未明記）。本番運用前に下記Critical 4件・High 6件の修正を必須とする。

---

## Critical（本番前修正必須）

### C-1. CRM運用方針の不整合（Notion → Supabase 未反映）
- **箇所**: `業務フロー定義.md` L188 「Notion CRMデータベース（予定）」
- **問題**: 直近の確定事項では CRM は Supabase 運用方針。本書では未反映で、過去方針が残存。
- **推奨修正**: L188を「リード〜提案：Supabase（顧客テーブル・案件テーブル）」に変更。Notion表記を全削除。

### C-2. frontmatter status のバージョン不整合
- **箇所**: `業務フロー定義.md` L5 `status: v1.0`
- **問題**: 改訂履歴ではv1.1まで進んでいるのに、frontmatterはv1.0のまま。会社資産として運用する際の版管理が壊れる。
- **推奨修正**: L5 を `status: v1.1` に更新。今後改訂のたびにL5と改訂履歴の両方を更新するルールをドキュメント末尾に追記。

### C-3. ¥30,000有償要件定義の表現が「成約時相殺」のまま（法的指摘未反映）
- **箇所**: `業務フロー定義.md` L90 「¥30,000有償要件定義（成約時相殺）」、メモリL17
- **問題**: ip-agent指摘で「相殺」は法的に不正確、「初期費用への充当」が正確と既に確定している。本書は未反映。
- **推奨修正**: L90を「¥30,000有償要件定義（成約時、初期費用に充当）」に変更。同一表現がメモリ`project_consulting-flow-skip-rule.md` L17にも残るため同期修正。

### C-4. Pattern C 判定ルール「月2件カウント」の残骸
- **箇所**: `業務フロー定義.md` L131 「Pattern Bの月2件カウントには含めない」
- **問題**: v1.1で月2件ルールは廃止（1顧客1件・3ヶ月クールダウンに変更）された。Pattern C側に旧ルール残骸あり。
- **推奨修正**: L131を「Pattern Bの『1顧客1件・3ヶ月クールダウン』カウントには含めない（紹介経由は別枠）」に変更。

## High（本番前修正推奨）

### H-1. 無料版レポート方針（C案）が定義書に未反映
- **箇所**: `無料版vs有償版_定義.md` L83-87 / `業務フロー定義.md` Pattern A-2
- **問題**: 直近確定の「無料版C案＝仕組み記述3件OK・情報源で差別化」が定義書に反映されていない。L83-87は「具体的なAIツール名禁止」のままで、仕組み記述（API連携・自動化フロー）の3件記載許容が不明。
- **推奨修正**: 無料レポート構成の「2. AI活用が効果的な業務 TOP3」に「仕組みのアウトライン3件まで記載可。情報源（業界特有データ・補助金最新情報）で差別化」を明記。

### H-2. 着手金/残金比率の未定義
- **箇所**: `業務フロー定義.md` Pattern A-11, B-8, C-7（契約）/ 共通-7（検収）
- **問題**: 「着手金請求」「残金請求」と書かれているが比率が未定義。50/50か30/70かで顧客対応がブレる。
- **推奨修正**: 「着手金 50%・検収後残金 50%」を共通フロー直前に明記。例外条件（30万円未満は一括前払い等）も併記。

### H-3. 納品後3ヶ月の無償範囲がサイトと定義書で粒度不一致
- **箇所**: `業務フロー定義.md` L147,150 / `implementation.astro` L61
- **問題**: 業務フロー定義は「不具合修正は無償、機能追加・改善は保守または個別見積」と明記。サイトは「納品後3ヶ月間の不具合修正（無償）」のみで、機能追加が無償と誤解される余地あり。
- **推奨修正**: implementation.astro L61 を「納品後3ヶ月間の不具合修正（無償）／機能追加・改善は別途」に修正、または業務フロー定義から該当箇所を「外部公表用 1行サマリ」として書き出し、サイト・契約書・定義書の3箇所で文言を完全一致させる。

### H-4. ステップ数のサイト/フロー齟齬
- **箇所**: `free-diagnosis.astro` L24-29（3ステップ表示）/ `業務フロー定義.md` Pattern A-1〜5（5ステップ）
- **問題**: サイトは「フォーム→AI分析→レポート送付」の3ステップ。業務フローでは無料MTG含む5ステップ。サイト訪問者は「無料MTGがある」ことを知らずレポート受領で離脱しうる。
- **推奨修正**: free-diagnosis.astro のステップ表示に「4. 無料MTG（特典・希望者のみ）」を追加するか、フォーム送信完了画面 / 自動返信メールで無料MTG特典を強調。

### H-5. キャンセル・トラブル時のフォールバックが未定義
- **箇所**: `業務フロー定義.md` 全体
- **問題**: 「失注」フローはあるが、(a)入金後の顧客都合キャンセル、(b)構築途中の顧客連絡途絶、(c)検収不可（仕様乖離）、(d)顧客側起因の遅延 が未定義。
- **推奨修正**: 新セクション「キャンセル・トラブル時のフロー」を追加。返金規定（着手金は要件定義工数として返金不可、等）と契約書側の準拠条項リンクを明記。

### H-6. 個人情報・契約書送付タイミング・領収書発行ルール不在
- **箇所**: `業務フロー定義.md` 全体
- **問題**: フォーム取得した個人情報の保管・破棄ルール、契約書を電子締結（クラウドサイン等）か紙か、領収書発行（freee自動 or 手動）の取り決めが定義書に存在しない。
- **推奨修正**: 「事務手続き」セクションを追加。(a)個人情報は失注後12ヶ月で削除、(b)契約は電子締結（クラウドサイン）、(c)領収書はfreeeで自動発行、を最低限明記。

## Medium（運用安定後に整備）

### M-1. 用語の揺れ
- **箇所**: 全文
- **問題**:
  - 「導入支援」（業務フロー定義）／「実装支援」（メモリL11）／「自動化システム構築」（implementation.astro）が混在
  - 「ヒアリング」「キックオフMTG」「実装ヒアリング」「無料MTG」「提案MTG」「詳細レポートMTG」「仕様確定MTG」が並存し定義不明瞭
- **推奨修正**: 用語集セクションを末尾に追加。「導入支援」を正式名称に統一（implementation.astro はサイト都合上残してよいが業務フロー定義では使わない）。MTG種別は「無料MTG / 実装ヒアリング / 提案MTG / キックオフMTG（仕様確定）」の4種に整理。

### M-2. 「簡易事前フォーム」の中身が未定義
- **箇所**: Pattern B-4, Pattern C-4
- **問題**: 「IT環境フォーム」と書かれるのみで、有料レポート用フォーム（無料版vs有償版_定義.md L219-229）と同一かが不明。
- **推奨修正**: Pattern B/Cの簡易事前フォーム = 無料版vs有償版_定義のIT環境表（L222-228）と同一、と明記。

### M-3. 「Pattern B 履歴確認」の手段
- **箇所**: Pattern B-2 「CRMで確認」
- **問題**: CRMがSupabaseになる前提が未明示。何のフィールド（last_inquiry_date, status等）で判定するか不明。
- **推奨修正**: Supabase顧客テーブルの`last_free_hearing_at`列で3ヶ月判定する旨を脚注で追加。

### M-4. Claude Code担当工程のレビュー責任が曖昧
- **箇所**: Pattern A-2, A-9 / 共通-2,3
- **問題**: 「Claude Code + CEO」と書かれるがCEO監修の所要時間が見積もられていない（A-2「10分」のみ無料版vs有償版_定義に記載、業務フロー定義には未転記）。
- **推奨修正**: 各ステップの所要に「Claude Code初稿○分 + CEO監修○分」の内訳を入れる。

## Low（任意）

### L-1. CTOコンタクト先の改訂日表記
- 改訂履歴（L199-200）が両方とも `2026-04-27` で同一日に2回改訂が並ぶ。バージョン履歴を見たとき「同日2回」が直感的でないので、`2026-04-27 (午前)` `2026-04-27 (午後)` 等にしておくと運用上わかりやすい。

### L-2. 「Pattern B 失注時は経費処理」の会計指示
- L103「営業活動の経費として処理」は会計仕訳の取り決めではないため、CFOと連携して「販売費 / 営業活動費」のどの勘定科目に乗せるかを別途決めて脚注に書くと事故が減る。

### L-3. 保守の「月次稼働確認」が頻度2行に重複
- L159「頻度: 月1回」と「出力: 稼働ログレビュー / 月1回」のセル内重複あり。表構造としては「頻度=月1回」「出力=稼働ログレビュー」が正しい。
