# 業務フロー定義 v1.1 — 死角・矛盾・想定漏れ徹底検出レビュー

## レビュー所感（5行）
v1.1は「ハッピーパス」を綺麗に描いているが、悪意ある顧客・契約紛争・一人法人特有の崩壊経路・基盤AIの停止・成果物の知財帰属など、「想定外」の入口が概ね無防備。  
特にPattern B/Cで省略された¥5,000フィルタリングが、本来排除すべきだった顧客タイプ（持ち逃げ・値切り・無限改修要求）への防壁を同時に外している点が看過されている。  
ip-agent指摘の要弁護士確認事項（消費税・特商法・著作権）が業務フローの「契約フェーズ」と接続されておらず、運用しながら法的グレーを積み上げる構造になっている。  
基盤がAnthropic（Claude Code）の単一依存である事実上のインフラリスクが、業務フローの中で1度も言及されていない。  
v1.2で「失敗シナリオ集（Failure Mode Catalog）」をフロー定義と並列の正本資料として位置付けることを強く推奨する。

---

## 最重要3つの死角（Top Critical）

### #1【致命的】Claude Code基盤の単一障害点が業務フローに全く反映されていない
ステップ2/9/6/5全てで「Claude Code自動生成」が前提。Anthropic側の障害・価格急騰・規約変更・国別規制（日本リージョン制限等）が起きた瞬間、Pattern A/B/C全部が同時に止まる。**事業継続計画（BCP）の概念がフロー定義に存在しない**。月500円〜1タスク0.01円の価格表現を顧客提案に組み込んでいる以上、API価格が2倍になるだけで提案ROIが全部破綻する。

### #2【致命的】成果物・提案書・アーキテクチャ図の「持ち逃げ」防御がゼロ
有料レポート（5-8p）・提案書PDF（3-5p）・アーキテクチャ図は、顧客が「自分でやる」「他社にやらせる」「社内エンジニアに渡す」だけで完結できるレベルの情報量。著作権帰属・転用禁止・第三者開示禁止・サンプルとしての改変利用範囲が業務フローのどこにも書かれていない。ip-agent報告でも「不成約時の成果物（要件定義レポート）の著作権帰属」は要弁護士確認事項として残っている。**現状は契約書テンプレートが間に合っていないのに有料レポートを納品する構造**。

### #3【致命的】一人法人で代表が稼働不能になった瞬間、全顧客が同時被害
Pattern A/B/C全工程の担当が「CEO」一択。代表が病気・事故・メンタル離脱した場合、月次稼働確認・チャット対応・トラブル対応・保守契約・無償サポート期間（3ヶ月）の全てが停止する。複数の保守顧客が同時にSLA相当の被害を受け、信用失墜→全顧客流出という一人法人特有の連鎖崩壊が定義書に1行も書かれていない。「**代理対応者・緊急時連絡規定・最低稼働保証の上限**」が契約書側で明記されていない場合、顧客側から損害賠償請求の余地を残す。

## 死角カタログ（致命的→注意の順、計18件）

### 致命的（事業継続を脅かすレベル）

#### D-01 基盤AIの単一依存（Top #1再掲）
- シナリオ: Anthropic障害・価格2倍化・地域規制・規約変更
- 影響度: 全Pattern停止 / 全保守契約毀損
- 発生確率: 中（過去3年内に実例あり、AWS東京リージョン障害級は数年に1回）
- 対策: マルチプロバイダ対応設計（GPT/Gemini/ローカルLLM）、API価格改定時の見積再交渉条項を契約書に明記

#### D-02 成果物持ち逃げリスク（Top #2再掲）
- シナリオ: 有料レポート¥5,000支払い→アーキテクチャ図を社内エンジニアに渡して内製化
- 影響度: 数十〜数百万案件の喪失 / 競合への情報流出
- 発生確率: 中〜高（B2Bで頻発する典型パターン）
- 対策: 有料レポート/提案書に「Optiens独占的設計に基づくものとし、第三者への開示・複製・転用を禁ずる」秘密保持条項。透かし入りPDF。著作権帰属はOptiens、利用許諾はOptiensとの本契約締結を条件と明記

#### D-03 一人法人の稼働不能リスク（Top #3再掲）
- シナリオ: 代表の入院・事故・メンタル離脱
- 影響度: 全保守顧客同時被害 / 信用失墜の連鎖
- 発生確率: 低だが排除不能（一人法人の構造的リスク）
- 対策: 契約書に「不可抗力条項」「3営業日以内応答SLA」「最大対応工数月X時間」を明記。緊急時連絡先（外部協力者）の事前整備。保守ライト/スタンダードの責任上限を金額で明記

#### D-04 セキュリティインシデント時の責任範囲が未定義
- シナリオ: Optiensが構築したAIエージェント経由で顧客の顧客情報・社内文書が漏洩（reference_security-incidents.md事例の.env漏洩級）
- 影響度: 個人情報保護法違反責任 / 損害賠償リスク
- 発生確率: 中（プロンプトインジェクションは現実の脅威）
- 対策: 契約書に「セキュリティ責任分界点」「賠償上限を契約金額の○倍まで」を明記。サイバー保険検討。納品時に脅威モデルを書面で渡す

### 重大（事業に強い悪影響）

#### D-05 ip-agent要弁護士確認事項が業務フローと接続されていない
- シナリオ: 消費税処理（充当 vs 値引き）、特商法B2B限定表記、不成約レポートの著作権が未確定のまま運用開始
- 影響度: 税務リスク / 行政指導 / 著作権紛争
- 発生確率: 中
- 対策: 契約書テンプレート完成・税理士確認・特商法表記サイト掲載を「Pattern稼働の前提条件」として業務フロー定義に記載

#### D-06 個人情報保護法対応の不在
- シナリオ: フォーム入力データ・ヒアリング録音・顧客の社内文書をOptiens管理下で保管。プライバシーポリシーが業務フロー定義から参照されていない
- 影響度: 漏洩時の報告義務違反 / 行政処分
- 発生確率: 中
- 対策: フォーム送信前同意・保管期限・削除請求対応をフロー化。ヒアリング録音の取り扱い規定。Notion CRM格納時の暗号化・アクセス権限

#### D-07 検収基準の曖昧さ（無限改修地獄）
- シナリオ: 共通フローのステップ7「顧客検収」に客観基準がない。顧客が「動かない」「使いにくい」と言い続けて残金支払いを拒否
- 影響度: キャッシュフロー悪化 / 工数膨張
- 発生確率: 高（受託開発の典型紛争）
- 対策: 検収基準を契約書に「テストケース合格率○%」「主要ユースケース3件動作」等の客観条件で定義。検収みなし条項（納品後10営業日以内に異議なき場合は検収完了）

#### D-08 検収後3ヶ月無償サポートの範囲が無制限
- シナリオ: 「不具合修正」の定義が曖昧。顧客が「これは仕様の不具合」と主張して機能追加を無償要求
- 影響度: 工数大幅増 / 利益喪失
- 発生確率: 高
- 対策: 「不具合 = 仕様書記載の動作と異なるもの」「環境起因（顧客側API変更）は対象外」と契約書定義。月次工数上限

#### D-09 顧客側従業員退職によるエージェント孤児化
- シナリオ: 担当者が退職→新担当者が運用方法を知らない→「使えない」とクレーム→保守解約
- 影響度: 保守契約喪失 / 売上計画毀損
- 発生確率: 高（中小企業の人材流動性は高い）
- 対策: 納品時に運用マニュアル + 動画記録を必須化。顧客側「運用責任者」の契約書明記。引き継ぎサポートを別料金オプション化

#### D-10 顧客のクラウドサービスへのOptiensアクセス権限の取り扱い
- シナリオ: Google Workspace / Slack / freee等のAPI連携でOptiensが顧客環境に入る。退任後の権限剥奪手順未定義
- 影響度: 不正アクセス疑義 / 退任後の責任問題
- 発生確率: 中
- 対策: 契約書に「アクセス権付与・剥奪手順」「契約終了時のキー・トークン削除責任」を明記。顧客側でOptiens用専用アカウントを発行する設計

### 重要（運用に支障）

#### D-11 Pattern B「3ヶ月クールダウン」の判定難
- シナリオ: 顧客「前回と違う案件です」と主張して再度無料ヒアリング要求
- 影響度: 工数の無料消化 / Pattern B運用形骸化
- 発生確率: 中
- 対策: 「同一顧客（法人格単位）」と明文化。CRMで履歴一元管理。再来時の判定権はCEOに留保する旨を契約書面で示す

#### D-12 無料診断フォームのDoS的悪用
- シナリオ: 競合・愉快犯が大量送信。無料レポート生成API課金の浪費・受信ノイズで本来の見込み客を見落とす
- 影響度: API課金浪費 / 機会損失
- 発生確率: 中（IT系では頻発）
- 対策: reCAPTCHA・レート制限・IP制限・メール認証必須化。フォーム→自動レポート生成の間に手動承認を1段階置く

#### D-13 紹介者経由トラブル時の関係毀損
- シナリオ: Pattern Cでトラブル発生→顧客が紹介者に苦情→紹介者がOptiensから離反
- 影響度: 紹介ルート全停止 / 信用毀損
- 発生確率: 中
- 対策: 紹介者契約に「紹介者免責」「Optiens直接対応原則」を明記。トラブル発生時の紹介者通知タイミングをフロー化

#### D-14 単発スポット対応（¥12,000/h）の積み上げ炎上
- シナリオ: 保守契約に進まない顧客が「ちょっとだけ」を繰り返し→月10時間消費→請求12万→トラブル
- 影響度: 工数圧迫 / 請求紛争
- 発生確率: 中
- 対策: スポット対応の事前見積必須化（30分超は書面合意）。月間スポット工数上限の設定。「保守契約への移行を強く推奨」する閾値を明記

#### D-15 倫理的グレー業務の依頼への対応規定不在
- シナリオ: 顧客が「競合の口コミを大量に書かせたい」「無断スクレイピングで競合価格収集したい」を依頼
- 影響度: 法的リスク / レピュテーション毀損
- 発生確率: 低〜中
- 対策: 契約書/利用規約に「禁止業務リスト」を明記。受注前ヒアリングで意図を確認するチェック項目を業務フローに追加

#### D-16 同時案件3件並走時のキャパ崩壊
- シナリオ: Pattern A納品+Pattern B提案+Pattern C設計が重なる週、所要日数が全部1.5倍に
- 影響度: 全顧客遅延 / 信用毀損
- 発生確率: 高（成功するほど起きる）
- 対策: 業務フロー定義に「同時並走上限N件」を明記。受注時に「着手可能日」を必ず提示する条項。Pattern別ではなく総工数で管理

### 注意（中長期で顕在化）

#### D-17 営業ピークと納品ピークの重複
- シナリオ: 紹介集中月に提案MTGが詰まり、同時に既存案件納品も重複
- 影響度: 提案品質低下 / 受注機会逸失
- 発生確率: 中
- 対策: 月次の「営業可動枠」「納品可動枠」を別管理。Notion CRMにキャパシティ可視化

#### D-18 メンタルヘルス・燃え尽きリスク
- シナリオ: ワンオペで休めない構造。受注プレッシャー＋納品プレッシャー＋顧客対応の常時オン状態
- 影響度: 一発で全事業停止
- 発生確率: 中（一人法人の長期経営で典型）
- 対策: 業務フロー定義に「営業停止期間（年間Nヶ月）」を制度化。LINE/メール対応の応答時間SLAを「3営業日以内」に明記し常時オンを否定

## 「想定外をどう想定するか」フレームワーク提案（v1.2向け）

### Failure Mode Catalog 方式の導入
業務フロー定義（v1.x）を「ハッピーパス」、Failure Mode Catalog（FMC v1.x）を「異常系」として2本立てで運用する。

#### FMCの3層構造
1. **L1: 顧客起因の異常**（持ち逃げ・無限改修・倫理グレー依頼・大量フォーム荒らし・担当退職）
2. **L2: Optiens起因の異常**（代表稼働不能・キャパ崩壊・納品遅延・セキュリティインシデント）
3. **L3: 外部起因の異常**（Anthropic障害・API価格改定・MCP仕様変更・法改正・サイバー攻撃）

#### 各異常項目に必須記載
- トリガー条件（何が起きたら発動するか）
- 一次対応（24時間以内）/ 恒久対応（1週間以内）
- 顧客への通知文テンプレート
- 損失上限（最悪ケースの金額・期間）
- 予防策（フロー定義 v1.x への反映）

### Pre-Mortem演習の定例化
- 月1回「この事業が3ヶ月後に死ぬとしたら何が原因か」を5分で書き出す（Pre-Mortem）
- 出てきた死因をFMCに反映
- 経営定例（コズロフ報告）の固定アジェンダにする

### v1.2 改訂時の必須追加項目
1. 全Patternに「契約書テンプレート完備」「特商法表記済」「税理士確認済」をDoR（Definition of Ready）として明記
2. 共通：導入支援フローに「セキュリティ設計レビュー（簡易脅威モデル）」ステップを追加
3. 共通：保守フローに「四半期1回のキャパシティ棚卸」「年1回の契約書改訂レビュー」を追加
4. 「営業可動枠」「納品可動枠」の月次上限を数値で記載（例: 提案MTG月X件、新規納品月Y件まで）
5. Failure Mode Catalogへの参照リンクを各フェーズに付与

## まとめ：v1.1の構造的弱点

業務フロー定義 v1.1は「事業が順調に進んだ場合の手順書」としては合格だが、**「事業を殺す経路」への防御**が体系化されていない。  
特に致命的なのは、(1) Claude Code基盤の単一依存、(2) 成果物の知財・転用防御、(3) 一人法人の稼働不能リスク の3点で、これらは契約書・BCP・サンドボックス設計と一体で対応しなければ、最初の本番案件で炎上する可能性が現実的にある。  
v1.2で「Failure Mode Catalog」を正本資料化し、契約書テンプレート完備をPattern稼働の前提条件として明記することを強く推奨する。
