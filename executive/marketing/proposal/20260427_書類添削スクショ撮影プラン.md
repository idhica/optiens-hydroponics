# 書類添削スクショ撮影プラン

## 目的

合同会社Optiensのサイト（AI支援事業ページ・ブログ・サンプル事例）に掲載する「経営者向け書類添削サービス」のスクリーンショット素材を、CEOがスマホで撮影できる状態にする。

ターゲットは、時間がない経営者・PCをあまり使わない経営者。AIエージェントによる文書レビュー支援が「スマホで完結する」体験を、視覚的に伝えるための素材。

## 既存資産の構造分析（novel-review.astro / API）

### フロントエンド（`src/pages/novel-review.astro`）
- 3カラム構成: 行番号 | 本文 | 修正メモ（textarea）
- 上部ツールバー: 作品セレクト・章セレクト・状態表示・クリアボタン
- スマホ対応CSS（@media max-width: 768px / 480px）あり、現状でもスマホ閲覧は可能
- メモ入力で2秒デバウンス保存。メモあり行は左端の行番号がアクセントカラー（赤）に変化
- 本文ソース: `/novels/<作品名>/<ファイル名>.md`（frontmatterはクライアントで除去）
- 章一覧ソース: `/novels/index.json`

### バックエンド（`src/pages/api/novel-review.ts`）
- GET: `key`をクエリパラメータで受け、Supabase `novel_review_notes` テーブルから notes JSON を返す
- POST: `{ key, title, notes }` を受け、同テーブルへ upsert + Google Tasks「AI小説事業」リストへ「推敲メモ反映: <title>」タスク自動作成
- `key` は `${novel}/${file}` 形式で一意化（書類用途でも同一テーブルで運用可能）

### sync-novels.mjs
- `c:/workspace/optiens-novel/projects/<作品>/chapters/*.md` を `public/novels/` にコピーし、`index.json` を生成
- 書類版でも同等のスクリプトを用意することで素材投入を自動化できる

## 案A / 案B / 案C 比較

| 観点 | 案A: novel_review_notes に書類データ投入 | 案B: /document-review 専用ページ新設 | 案C: novel-review に mode 切替 |
|---|---|---|---|
| 実装工数 | 最小（フロント1箇所＋index.json差し替え） | 中（ページ複製＋API分岐 or 流用） | 中（クエリパラメータ判定の分岐ロジック追加） |
| スクショ品質 | タイトルが「推敲ツール」のままだと違和感 | 「書類添削」専用UIで違和感なし | mode判定ロジックでタイトル変更可能 |
| Supabaseの整合性 | 小説データと混在（key prefix で分離可能） | 別テーブル（document_review_notes）が綺麗 | 同テーブル混在 |
| Google Tasks連携 | 「AI小説事業」リストにタスクが流れる（誤動線） | 連携を切る or 別リスト指定 | mode別に分岐 |
| 将来の拡張 | 兼用ツールとしてやや無理が出る | 書類添削サービスへ昇格しやすい | コードが複雑化 |
| 公開範囲 | 同じURL（`/novel-review`）でアクセス | 新URL（`/document-review`） | 同じURL＋クエリ |

## 推奨: 案B（/document-review 専用ページ新設）

### 推奨理由
1. **スクショ品質が最優先**: 撮影した画像をAI支援事業の事例コンテンツに使う以上、「推敲ツール」と表示されたまま使うのは不自然。専用UIなら「経営者向け書類添削」「AIによる指摘メモ」など、サービスとしての世界観を直接伝えられる。
2. **将来の事業化に直結**: AI支援事業のロードマップ（Phase A2「診断レポート自動生成の仕組み化」、A3「IoT提案」）と整合する。書類添削は経営者の入口体験として有力で、専用ページがあれば顧客に直接見せられる。
3. **既存novel-reviewへの副作用ゼロ**: 小説事業のメモ運用（Google Tasks「AI小説事業」リスト連携）を壊さずに済む。
4. **データの分離**: 小説メモと書類メモが同テーブルに混ざらず、後の集計・KPIで困らない。
5. **実装工数は中程度**: novel-review.astroをコピーして用語と読み込み元を差し替えるだけで原型ができる。Supabaseテーブルは novel_review_notes の構造をそのまま流用（テーブル名のみ document_review_notes）。

### 必要な実装変更（最小構成、CEO承認後に着手）

1. **新ページ**: `src/pages/document-review.astro`（novel-review.astroをベースに用語差し替え）
   - タイトル: 「書類添削ツール | Optiens」
   - ツールバー: 「推敲ツール」→「書類添削」、「作品を選択」→「書類を選択」、「章を選択」→（撤去 or 単一階層に）
   - カラムヘッダー: 「本文」「修正メモ」→「本文」「AI添削メモ」
   - メモ欄を読み取り専用にせず、CEOが追記・編集可能な現行仕様を維持（撮影時の自由度を保つ）

2. **新API**: `src/pages/api/document-review.ts`（novel-review.tsをコピー）
   - テーブル名を `document_review_notes` に変更
   - Google Tasks 連携は撮影目的では不要 → 一旦コメントアウト or 別タスクリスト指定

3. **新Supabaseテーブル**: `document_review_notes`（既存と同スキーマ）
   ```sql
   create table document_review_notes (
     key text primary key,
     title text not null,
     notes jsonb not null default '{}',
     note_count integer default 0,
     updated_at timestamptz default now()
   );
   ```

4. **素材ファイル投入**:
   - `public/documents/01-apology-mail.md` 〜 `07-presidents-greeting.md` を配置（frontmatter付きMarkdown）
   - `public/documents/index.json` を手動生成 or 簡易syncスクリプトで生成
   ```json
   [
     { "file": "01-apology-mail.md", "title": "取引先へのお詫びメール" },
     { "file": "02-contract-draft.md", "title": "業務委託契約書ドラフト" },
     { "file": "03-internal-notice.md", "title": "社内通達（リモートワーク制度変更）" },
     { "file": "04-press-release.md", "title": "プレスリリースドラフト" },
     { "file": "05-customer-notice.md", "title": "顧客向け案内メール（料金改定）" },
     { "file": "06-job-posting.md", "title": "求人票（営業職）" },
     { "file": "07-presidents-greeting.md", "title": "社長挨拶文（年頭所感）" }
   ]
   ```

5. **Markdownファイルの構造変換**:
   現状のサンプルMarkdownは「書類本文」「添削メモ」の2セクションを1ファイルに併記している。document-review上で扱う際は、本文部分のみを読み込み対象とし、添削メモは事前にSupabaseへ投入する形にする。
   - 本文と添削メモの分離スクリプトを用意（後述「データ投入手順」参照）

## スマホスクショ撮影プラン

### 機材
- CEOのiPhone（既存運用と同じ）
- ブラウザ: Safari（広告なし・URL欄を非表示にできる「リーダー外」表示）
- ステータスバー（時刻・電池）が綺麗な時間帯（毎時00分・電池満充電）に撮影推奨

### 撮影手順（案B採用時）
1. iPhoneで `https://optiens.com/document-review` にアクセス
2. 書類セレクタで「取引先へのお詫びメール」を選択
3. 事前投入済みの添削メモが3〜5件、メモあり行が赤強調で表示される
4. CEOが任意の行をタップ→キーボードを出さずスクリーンショット撮影
   （iPhone: サイドボタン+音量上ボタン同時押し）
5. 続けて、メモ欄をタップしてキーボードが出た状態でも1枚撮影
   （「スマホで実際に編集できる」ことを示す素材）
6. 7書類分すべて、もしくは代表3〜4書類分を撮影
7. 撮影後、不要なキャッシュ・履歴は残さない（公開素材として安全）

### 撮影バリエーション（推奨7パターン）
| # | 構図 | 用途 |
|---|---|---|
| 1 | お詫びメール 全体表示 | LP「短時間でレビュー」訴求 |
| 2 | お詫びメール メモあり行をタップ | LP「指摘箇所を即確認」訴求 |
| 3 | 契約書ドラフト 高重要度メモ表示 | ブログ「契約書レビュー事例」 |
| 4 | プレスリリース 過大表現指摘 | ブログ「広報文書のAIレビュー」 |
| 5 | 求人票 業務範囲指摘 | LP「採用文書もレビュー対象」 |
| 6 | 顧客向け案内メール 値上げ通知 | LP「炎上リスクを事前に検知」 |
| 7 | 社長挨拶文 抽象表現指摘 | ブログ「経営者の発信を整える」 |

### 撮影時の注意
- 個人情報・実在企業名が画面に映っていないこと（架空企業名のみで作成済み）
- ブラウザのオートフィル候補が表示されていないこと（プライベートブラウズ推奨）
- スマホ通知が画面上部に表示されていないこと（おやすみモード推奨）

## データ投入手順（案B採用時）

### Step 1: 本文と添削メモの分離（作業者: CEO or technical-writer）
`executive/ai-consulting/sample-documents/*.md` の各ファイルから:
- 「## 書類本文」セクションを抽出 → `public/documents/<番号>-<種別>.md` として保存（frontmatter付き）
- 「## 添削メモ（AI提案）」セクションを抽出 → 行番号にマップしたJSON形式に変換 → Supabase `document_review_notes` に投入

### Step 2: 行番号マッピング
書類本文を表示する際、`document-review.astro` 側で空行も含めて1行=1配列要素になる。添削メモの「箇所」を該当行番号に紐づける必要がある。
- 投入時に各書類ごとに行番号と添削メモを手動マッピングする
- 例: 「敬語重複でくどい」→ 該当する本文行番号（例: 11行目）にメモを紐付け

### Step 3: Supabase投入
Supabase ダッシュボード or REST APIで以下を投入:
```json
{
  "key": "documents/01-apology-mail.md",
  "title": "取引先へのお詫びメール",
  "notes": {
    "11": "【重要度: low】「大変ご迷惑をおかけしておりますこと、まずは深くお詫び申し上げます」: 敬語重複でくどい。\n提案: 「ご迷惑をおかけしておりますこと、深くお詫び申し上げます」に簡潔化。",
    "14": "【重要度: high】原因説明が弱い。\n提案: 「材料手配の発注確認漏れにより、製造着手が3営業日遅れたためです」のように具体度を上げる。",
    "...": "..."
  },
  "note_count": 5
}
```

### Step 4: ローカル動作確認 → 本番反映
- `npm run dev` で `/document-review` にアクセスし、書類選択→メモ表示までを確認
- 問題なければ commit → push → Vercel自動デプロイ
- スマホで本番URLにアクセスし、撮影開始

## CEOへの確認事項

1. **推奨案B（専用ページ新設）で進めてよいか**
   - 「スクショ撮影が主目的・将来事業化を見据える」前提で案Bを推奨
   - 「とにかく早く撮影したい」が最優先なら案A（既存novel-reviewにデータ投入のみ）も選択肢

2. **書類7件のラインアップとリアリティの可否**
   - 1: お詫びメール / 2: 業務委託契約書 / 3: 社内通達 / 4: プレスリリース / 5: 顧客向け案内（料金改定） / 6: 求人票 / 7: 社長挨拶文
   - 業種・読み手のバランス: 取引先・社内・メディア・顧客・求職者・社員 を一通りカバー
   - 不要なものを差し替える、または不足するもの（例: 銀行融資依頼書、役員会議事録）を追加するかの判断

3. **添削メモの「重要度ラベル」を画面でどう見せるか**
   - 現案: メモ本文の冒頭に `【重要度: high】` を付与する素朴な方法
   - 別案: high/medium/low で行のハイライト色を変える（要CSS追加・実装工数増）

4. **公開URL運用**
   - 案B採用時、`/document-review` を一般公開とするか、`?token=xxx` のような簡易ガード付きにするか
   - 撮影目的のみなら本番ノーガード公開でOK。事業化フェーズで認証を入れる方針でよいか

5. **Google Tasks連携の扱い**
   - 撮影目的では連携不要。事業化フェーズで「保守タスク自動起票」などに転用できる
   - 一旦コメントアウト方針でよいか
