# 日本企業のAI導入におけるセキュリティ懸念 調査レポート

## 1. 日本企業のAI導入阻害要因の統計データ

### 1.1 総務省 令和7年版 情報通信白書（2025年7月公表）

企業における生成AI導入の懸念事項（公的データ）:

| 順位 | 懸念事項 | 備考 |
|------|---------|------|
| 1 | 効果的な活用方法がわからない | 最多回答 |
| 2 | 社内情報の漏洩等のセキュリティリスク | 2番目に多い |
| 3 | ランニングコストがかかる | |
| 4 | 初期コストがかかる | |

出典: https://www.soumu.go.jp/johotsusintokei/whitepaper/ja/r07/html/nd112220.html

### 1.2 JIPDEC「企業IT利活用動向調査2026」（2026年3月公表）

AI導入における課題の構造（公的データ）:

**導入前の課題（上位）:**
- AIの開発や運用ができる人材やスキルの不足
- 従業員のAIに関する教育やリテラシーの不足
- 活用目的や効果指標の不明確さ

**導入後も残存する課題:**
- AIの学習や利用時のセキュリティ対策の懸念
- AIの学習のためにデータ化されていない情報が多い
- 個人情報や機密情報をAIで扱う際のプライバシーの懸念
- AIの出力結果に対する信頼性や精度への不安

出典: https://www.jipdec.or.jp/library/it-resarch/it-resarch2026-02.html

> 重要な示唆: セキュリティ・プライバシー・信頼性の懸念は「導入すれば解消される」類のものではなく、導入後も継続する構造的課題。

### 1.3 NRI「IT活用実態調査2025」

- 生成AI導入済み企業: 57.7%（2023年度33.8% → 2024年度44.8% → 2025年度57.7%）（公的データ）
- リスク管理が難しいと回答: 48.5%（公的データ）
- リテラシー・スキル不足と回答: 70.3%（公的データ）

出典: https://aismiley.co.jp/ai_news/nri-it-2025-ai/

### 1.4 ガートナージャパン調査（2025年3月）

- 生成AIサービス利用企業: 約63%（公的データ）
- 管理ルールを整備している企業: 約2割（公的データ）

> 示唆: 6割超の企業が利用しているにもかかわらず、ガバナンス体制が追いついていない。これがセキュリティ懸念の根本原因。

### 1.5 PwC「生成AI実態調査2025春」5カ国比較

- 脅威認識が「コンプライアンス・企業文化における脅威」に44%がシフト（前回比+23pt）（公的データ）
- 日本企業はAI活用の「成果実感」が他4カ国に比べて著しく低い

出典: https://www.pwc.com/jp/ja/knowledge/thoughtleadership/generative-ai-survey2025.html

## 2. 具体的なセキュリティ懸念の内訳

### 2.1 機密データの外部送信リスク

**懸念の内容:**
- プロンプトに含まれる社内機密情報（売上データ、顧客情報、戦略文書）が外部サーバーに送信される
- クラウド上で処理されるため、通信経路上での漏洩リスク
- サービス提供企業の社員がデータにアクセスする可能性

**実態:**
- 主要AIサービスはいずれもAES-256暗号化（保存時）、TLS 1.2+（通信時）を採用
- Enterprise契約ではデータアクセス制限が明確に規定されている

### 2.2 AIモデルへの学習利用

**懸念の内容:**
- 入力したデータがAIモデルの改善・学習に使われ、他のユーザーへの回答に反映される
- 競合企業が同じサービスを使った場合、自社の情報が間接的に漏洩する

**実態:**
- Enterprise/API契約では各社とも「学習に使用しない」ことを明示
- 無料・個人プランでは学習利用される場合がある（後述の比較表参照）

### 2.3 プロンプトインジェクション・情報漏洩

**懸念の内容:**
- 悪意あるプロンプトにより、システムが意図しない情報を出力する
- 社内AIシステムに対する攻撃で機密情報が引き出される

**実態:**
- IPA「情報セキュリティ10大脅威 2026」で「AIの利用をめぐるサイバーリスク」が初選出（公的データ）
- 既存の攻撃がAIによって質的に変化している段階

出典: https://www.trendmicro.com/ja_jp/jp-security/25/l/expertview-20251225-01.html

### 2.4 生成結果の著作権・責任問題

**懸念の内容:**
- AIが生成したコンテンツの著作権帰属が不明確
- AI出力に基づく業務判断で損害が生じた場合の責任所在
- AIが他者の著作物を無断で再現するリスク

**実態:**
- 日本の著作権法では「AI生成物に人間の創作的寄与がある場合」は著作物として保護される方向
- 各社の利用規約ではAI出力の最終責任はユーザーに帰属

### 3.2 各サービスの特記事項

**Claude (Anthropic):**
- Team/Enterprise/APIプランでは入力データをモデル学習に使用しない
- Free/Proプランは2025年のポリシー変更でオプトアウト方式に移行（デフォルトで学習に利用される）
- 2026年H1にBYOK（Bring Your Own Key）対応予定
- SOC 2 Type IIレポートはNDA下でTrust Portalから閲覧可能

出典: https://code.claude.com/docs/ja/data-usage / https://ensou.app/blog/anthropic-claude-terms-and-privacy-update-2025/

**ChatGPT (OpenAI):**
- Enterprise/Businessプランはデフォルトで学習利用しない
- Plusプランは設定 > Data Controlsで手動オプトアウトが必要（デフォルトは学習に利用）
- Enterprise契約でカスタムデータ保持期間設定可能（最低90日）
- データリージョン選択が可能（日本含む）

出典: https://help.openai.com/en/articles/7730893-data-controls-faq

**Microsoft Copilot:**
- Microsoft 365のテナント境界内でデータ処理（Azure OpenAI経由、OpenAI公開サービスとは別基盤）
- 既存のMicrosoft 365セキュリティポリシー（DLP、感度ラベル、保持ポリシー）がそのまま適用
- ユーザーのアクセス権限を継承（過剰なアクセス権限がある場合はリスクになる点に注意）
- 2026年1月よりAnthropicがサブプロセッサーとして追加

出典: https://learn.microsoft.com/en-us/copilot/microsoft-365/enterprise-data-protection

**Google Gemini:**
- Workspace契約企業のデータはモデル学習に使用しない、人間によるレビューもなし
- VPC Service Controls、CMEK（Customer Managed Encryption Keys）、EKM対応
- DLPポリシーでIRM（Information Rights Management）が適用されたファイルはGeminiの検索対象外
- FedRAMP High認証取得

出典: https://docs.cloud.google.com/gemini/docs/discover/data-governance

## 4. 反論ロジック（AI導入提案時の想定問答）

### 4.1 「データが外部に漏れるのでは？」

**反論:**
- 主要AIサービスはいずれも通信時TLS 1.2+、保存時AES-256で暗号化されており、銀行のオンラインバンキングと同等以上のセキュリティ水準
- Enterprise契約ではデータへのアクセスが厳格に制限され、SOC 2 Type II認証で第三者監査済み
- Microsoft CopilotはMicrosoft 365テナント内で処理されるため、既存のセキュリティ境界がそのまま適用される
- 「外部送信」という表現は正確ではない。暗号化された通信経路で認証済みのサーバーに送信され、処理後は保持期間を経て削除される

### 4.2 「入力データが学習に使われるのでは？」

**反論:**
- Enterprise/API契約では全主要サービスが「学習に利用しない」と契約上明示している
- 無料プランでも、OpenAIは設定画面からオプトアウト可能
- Claude APIは入力データをモデルトレーニングに使用しないことが利用規約に明記
- 法人契約（Team/Enterprise）を選択すれば、この懸念は契約レベルで解消される
- 月額数千円〜のTeamプランでも学習除外が適用されるため、コスト障壁は低い

### 4.3 「社内の機密情報をAIに入れてよいのか？」

**反論:**
- まず、全ての情報をAIに入れる必要はない。「機密度に応じた使い分け」が基本
- 段階的アプローチ: 一般的な業務文書 → 社外秘レベル → 機密レベルと段階的に範囲を拡大
- Enterprise契約ではZDR（ゼロデータリテンション）オプションがあり、処理後即時削除が可能
- Microsoft Copilotは既存の情報保護ポリシー（感度ラベル、DLP）をそのまま継承するため、現在のセキュリティ体制の延長線上で導入可能
- オンプレミスでの運用が必要な場合、Azure OpenAI Service等のプライベートクラウド選択肢もある

### 4.4 「AIの回答を業務判断に使って責任は？」

**反論:**
- AIはあくまで「補助ツール」であり、最終判断と責任は人間にある。これは全AIサービスの利用規約に明記
- JIPDECの調査でも、AIガバナンスで最も重視されるのは「人間による最終判断の確保」
- 実務的には「AI出力を参考にし、人間がレビュー・承認するワークフロー」を設計すればよい
- 医療・法律等の専門分野では、専門家の確認を必須とする運用ルールを設定する
- AI出力の根拠・判断過程を説明できる体制を整備することが重要（JIPDEC調査でも最優先課題として挙げられている）

## 5. 中小企業向け 現実的なセキュリティガイドライン

### 5.1 最低限守るべき5つのルール

| # | ルール | 具体的な実施方法 |
|---|--------|----------------|
| 1 | **個人情報を入力しない** | 氏名・住所・電話番号・マイナンバー等は入力前に必ず匿名化またはマスキング |
| 2 | **法人契約プランを使う** | 無料プランの業務利用は禁止。最低でもTeamプラン（学習除外）を契約 |
| 3 | **機密レベルを分類する** | 社内情報を3段階（一般/社外秘/機密）に分類し、AIに入力可能な範囲を明確化 |
| 4 | **出力結果を鵜呑みにしない** | AI出力は必ず人間がファクトチェック。特に数値・法的判断・医療情報は専門家確認必須 |
| 5 | **利用ログを残す** | 誰が・いつ・何の目的でAIを利用したかを記録する仕組みを導入 |

### 5.2 段階的導入アプローチ（3ステップ）

**Step 1: 試験導入（1-2ヶ月）**
- 対象: IT部門または特定チーム（3-5名）
- 用途: 議事録要約、メール文面作成、一般的な調査など低リスク業務のみ
- 個人情報・機密情報の入力は禁止
- 利用ガイドラインのドラフト作成

**Step 2: 限定展開（3-6ヶ月）**
- 対象を2-3部門に拡大
- 社外秘レベルの情報も対象に含める（Enterprise契約前提）
- 利用ガイドラインの正式策定・全社周知
- インシデント発生時の報告フローを整備

**Step 3: 全社展開（6ヶ月以降）**
- 全部門への展開
- 定期的なセキュリティ研修の実施
- 利用状況のモニタリングと改善
- 業務プロセスへの本格組み込み

### 5.3 参考: IPA「中小企業の情報セキュリティ対策ガイドライン」第4.0版（2026年3月公表）

IPAが公表した最新版ガイドラインでは、サイバーセキュリティ対策の記載が拡充されている。AI固有の項目はないが、以下の基本6か条は生成AI利用時にも適用される:

1. OSやソフトウェアは常に最新の状態にしよう
2. ウイルス対策ソフトを導入しよう
3. パスワードを強化しよう
4. 共有設定を見直そう
5. 脅威や攻撃の手口を知ろう
6. **バックアップを取ろう（第4.0版で新規追加）**

出典: https://www.ipa.go.jp/pressrelease/2025/press20260327.html

## 6. Optiens AI支援事業への示唆

### 6.1 提案時のポジショニング

セキュリティ懸念は「導入しない理由」として最も使われやすいが、実態としては:
- Enterprise/API契約を選択すれば、主要サービスはいずれも学習利用しない
- 暗号化・認証・監査の水準は金融機関が利用できるレベル
- 真の課題は「セキュリティそのもの」ではなく「正しい情報を知らないこと」

### 6.2 診断レポートへの組み込み

AI活用診断レポート（15万円）において、以下を標準セクションとして含めることを推奨:
- クライアント企業の情報分類（一般/社外秘/機密）の整理
- 推奨AIサービスと契約プランの選定理由
- セキュリティガイドラインのテンプレート提供
- 段階的導入スケジュールの提案

### 6.3 反論材料の活用場面

- 商工会でのセミナー・勉強会での説明資料として
- 診断レポートの「セキュリティ対策」セクションとして
- 経営者向け提案書の「よくある懸念への回答」として
